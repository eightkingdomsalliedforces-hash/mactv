# CSS1 搜尋效能設計

## 目標

CSS1 在進入選集頁時，同時搜尋所有尚可用的 web-selector 網站，避免慢站的等待時間逐站累加。搜尋結果、來源失敗原因與自動跳過失敗網站的既有行為必須保留。

## 根因

`AniSubsCSS1SubscriptionProvider.search` 目前用兩層串行迴圈：先逐站搜尋，再逐筆下載候選作品詳情。每次 HTTP 請求可等待 8 秒，因此多個慢站或多筆候選會讓總時間接近所有逾時總和，最後再被目錄層的 45 秒 CSS1 上限截斷。

## 設計

- 每個 CSS1 網站各自完成搜尋頁、候選過濾與詳情解析；不同網站以 Swift task group 並行。
- 同一網站最多解析前 6 筆相關候選，並以 task group 並行下載詳情頁，避免單站多候選線性等待，同時限制瞬間請求量。
- 每個網站回傳結果、失敗原因及健康狀態，不在並行工作內直接寫檔；聚合完成後依原始網站順序集中更新健康狀態，避免並行讀寫健康檔案互相覆蓋。
- 保留單次 HTTP 請求逾時、Bangumi 資料補全、繁簡比對、動畫類型過濾及完整 CSS1 失敗原因。
- 不縮短 8 秒請求逾時，以免把本來可用但稍慢的 CSS1 網站誤判成失敗。

## 驗證

新增兩個各延遲 150ms 的 CSS1 網站。舊串行實作至少約 300ms；並行後應在 260ms 內取得兩站結果。完整 `TVShellChecks` 必須通過。
